Theory

Definitions

A mixin is a class from which other classes can adopt methods and properties.

Mixins are used if you don't want a class to "inherit" from another class (i.e. be its child class) but you want it to "include" some attributes / methods.

You can think of mixins as uncles and aunts but not necessarily parents. They help avoid issues and complexities of multiple inheritance. e.g. If class TA has parents Student and Faculty, both of whose parent is Person, then does TA use Student or Faculty's version of any given method, or both's? (src: 1)

  • inherit functionality
  • mix/match functionality

Rationales

R1) A mixin is an object that we can use in order to add reusable functionality to another object or class, without using traditional inheritance. We can't use mixins on their own: their sole purpose is to add functionality to other objects or classes (src: 1, 2).

R2) Mixins can be used in this way by multiple classes, for code reusability. They are used to avoid code repetition and promote code reuse, so there is less complexity and room for error. This helps with collaboration.

R3) A mixin is a special kind of multiple inheritance. There are two main situations where mixins are used:

  • You want to provide a lot of optional features for a class.
  • You want to use one particular feature in a lot of different classes. (src: 1, 2)

R4) Another situation is: you want to provide a lot of (not-optional) features for a class, but you want the features in separate classes (and in separate modules) so each module is about one feature (behavior). In other words, not for re-use, but for compartmentalization. In this way, code at different levels (data touching, logic, view touching, etc.) is separated so different developers can collaborate easily. (src: 1, 2)

R5) Mixins can be used for adding lots of optional attributes/methods. You may want a class to avail of several optional properties or methods. (src: 1, 2)

R6) Mixins can be viewed as an interface with implemented methods. An interface is a protocol or a set of method signatures that is implemented in detail by some classes.

In this way, mixins can enforce the "dependency inversion principle", which states that high level interfaces and low level implementations should be decoupled, and each built based upon abstractions (i.e. the interface). High level objects should not depend on low level implementations (and vice versa). Either could be switched out, and as long as they respect the interface, the system should work. (src: 1 2 3 4 5 6)

R7) Multiple Versions of Methods / Props Note: When mixin A and mixin B each define the same method / property X, then an instance of class C, which adopts both mixins A and B, follows a method resolution order (MRO) to determine which version of X (A's or B's) to run. By using the super() syntax in A and B's version of the method / property X, one can even get both versions of X to run in sequence, one after another. This is a benefit of using super() with multiple inheritance, i.e. Python mixins (src: 1, 2, 3, 4, 5, 6, 7).

Mixins let a class adopt methods and attributes of another class. Reasons:
(A) code reuse
(B) optional methods / attributes
(C) compartmentalization

A mixin class acts as the parent class, containing the desired functionality. A subclass can then inherit or simply reuse this functionality, but not as a means of specialization. Typically, the mixin will export the desired functionality to a child class, without creating a rigid, single "is a" relationship. Here lies the important difference between the concepts of mixins and inheritance, in that the child class can still inherit all the features of the parent class, but, the semantics about the child "being a kind of" the parent need not be necessarily applied.

In this way also, mixins allow inheritance and use of only the desired features from the parent class, not necessarily all of the features from the parent class. Most languages don't give a mixin any special status; it's just a class that was designed to be "mixed in", rather than used standalone. It's a matter of the role of the class in the overall domain as to whether a mixin is a "primary" class or simply a mixin. You don't create objects / instances of the mixin classes but rather just mix the mixin classes' functionality into other classes.

(taken from src: 1, 2, 3)

Canonical / Tutorial Example

For an example, consider werkzeug, which is an HTTP request and response system. Werkzeug needs to support regular HTTP requests, but also:

  • requests that accept header support, which enables the system to retrieve all the relevant HTTP accept headers, that in turn determine the type of content the client can support
  • requests that support accept headers, etags (entity tags which help the web cache only request for information it needs), authentication, and user agent support (which lets servers access frontend client data such as the OS, browser, etc.)

Relevant Code

In Werkzeug, I can make a plain request object by saying:

    from werkzeug import BaseRequest
        class RegularRequest(BaseRequest):
            pass

If I want to add accept header support, which enables getting all HTTP accept headers (which determines the type of content the client can support) into a single object, I would instead write:

    from werkzeug import BaseRequest, AcceptMixin
    class SpecialRequest(AcceptMixin, BaseRequest):
        def sr_method:
            method_from_accept_mxn(prop_of_basereq_mxn)

If I wanted to make a request object that supports accept headers, etags (entity tags which help the web cache only request for information it needs), authentication, and user agent support (which lets servers access frontend client data such as the OS, browser, etc.), I could do this:

    from werkzeug import BaseRequest, AcceptMixin, ETagRequestMixin, UserAgentMixin, AuthenticationMixin

    class ExtraSpecialRequest(AcceptMixin, ETagRequestMixin, UserAgentMixin, AuthenticationMixin, BaseRequest):
        def esr_method:
            method_from_accept_mxn(prop_of_basereq_mxn)
            method_from_etag_mxn(prop_from_useragent_mxn)
            method_from_auth_mixin()
Rationales

R1 Ex) The difference is subtle, but in the werkzeug example, the mixin classes weren't made to stand on their own. In more traditional multiple inheritance, the AuthenticationMixin (for example) would probably be something more like Authenticator. That is, the class would probably be designed to stand on its own (src: 1, 2)

R2 Ex) Here werkzeug's SpecialRequest and ExtraSpecialRequest both use AcceptMixin.

R3 Ex) Here werkzeug is adding optional features for SpecialRequest and even more optional features for ExtraSpecialRequest.

R4 Ex) Here AcceptMixin, ETagRequestMixin, UserAgentMixin, AuthenticationMixin each represent compartmentalized implementations of particular functionalities.

R5 Ex) For instance in the werkzeug's request and response example, mixins let you allow more supports as needed, but not by default, to instances of Request. Adding mixins to classes let you layer on additional functionality as needed.

R6 Ex) Here, the werkzeug example's mixins (AcceptMixin, ETagRequestMixin, UserAgentMixin, AuthenticationMixin, etc.) respect DI (the dependency inversion principle) because each of their implementations could be changed as long as they implement the methods they promise to implement, which get used by the adopter classes (SpecialRequest, ExtraSpecialRequest, etc.). Similarly the adopter classes can change how they use the mixins and mixin methods they adopt depending only on the interface (i.e. the method signatures) of the individual mixins.

Summary

Werkzeug's Request / Response System Reasons:
(A) all Requests (SpecialRequest, ExtraSpecialRequest, etc) can use the AcceptMixin mixin
(B) the mixins let you allow more supports as needed, but not by default, to instances of the Request classes (i.e. AcceptMixin for SpecialRequest, 4 mixins for ExtraSpecialRequest)
(C) functionality for generally reusable request-related features (AcceptMixin, ETagRequestMixin, etc.) can be separated from ExtraSpecialRequest-specific functionality (i.e. the logic in the esr_method)

SpecialRequest, ExtraSpecialRequest, etc. are not types of AcceptMixin, ETagRequestMixin, UserAgentMixin, AuthenticationMixin, etc. The request classes ("children" / adopters) use methods of the mixin classes ("parents" / adoptees). The role of these mixins is to provide additional functionality and features to their adopter classes, and we do not create objects / instances of the Mixin classes.

Definitions

MV* (MVC, MVVM, MVP, MVT) divide user interface implementations into 3 interconnected elements - the model for data related management, the view (or in the case of django the "template") for visual representations, and the controller for logic for manipulating the model or view (src: 1). Nuances:

The model is responsible for managing the data of the application. It receives user input from the controller.

The view renders presentation of the model in a particular format.

The controller responds to the user input and performs interactions on the data model objects. The controller receives the input, optionally validates it and then passes the input to the model.

Rationales

R1) MV* affords separation of visualization, processing, and data management code.

R2) MV* affords modularity in code.

R3) MV* affords flexibility in code. For instance, one can change or add new tempalates (views) without changing or touching the underlying models.

R4) MV* affords testability in code. For instance, one can easily test the models and controllers with unit tests, and the templates (views) with e2e tests.

R5) MV* affords maintainability in code. One can more easily maintain smaller, separate chunks of code responsible for specific purposes (M, V, and C).

R1 Ex) The ToDoItem and User models, the CRUD operations on them, and the actual template (MV* view) for seeing the items on the page are in separate parts of the codebase, which makes it easier to separate visuals from processing from data.

R2 Ex) The ToDo list code is modular in that the ToDoItem objects can be reused by different controllers and templates.

R3 Ex) The ToDo list codebase is flexible. For example, one could add more models to store data like groups of lists and categories of items without affecting the existing functionality.

R4 Ex) The ToDo list codebase is testable. For example, e2e tests can evaluate template / view level ToDo list functionality and unit tests can evaluate the the controllers and models.

R5 Ex) The Todo list code is comprised of small chunked files and hence is easy to maintain.

No Framework Not using any framework. Mixing data touching, view touching logic like in vanilla PHP.

While there are many benefits of MV* over vanilla PHP, vanilla PHP is potentially easier for developers to learn, doesn't expose you to functionalities you don't need (as opposed to huge frameworks like CakePHP, Laravel, etc.), and is highly customizable. This is good for one-off scripts, demos, small API's, and highly specific use-cases (src: 1, 2).
